# Мастер-промпт для Claude Code: VivAtlas

Ты работаешь над проектом **VivAtlas** — самостоятельной self-hosted программой для каталогизации, поиска, объяснения и рекомендации скиллов, агентов, MCP-серверов, команд, плагинов, скриптов и других дополнений, хранящихся в приватных GitHub-, Gitea- и локальных Git-репозиториях.

## Режим работы

Сначала перейди в режим планирования и **не начинай реализацию сразу**.

1. Изучи текущий репозиторий целиком, но без бессмысленного чтения бинарных файлов, зависимостей и generated-каталогов.
2. Найди существующую архитектуру, документацию, предыдущий MVP, прототип Skill Librarian, конфигурации, тесты и уже реализованные компоненты.
3. Не считай проект пустым и не создавай параллельную архитектуру, пока не выяснишь, что уже существует.
4. Не меняй существующую структуру каталогов без необходимости.
5. Не удаляй и не переписывай рабочий код только ради выбранного тобой стиля.
6. Сначала подготовь полный план, список найденных компонентов и список блокирующих вопросов.
7. Задай вопросы одним компактным блоком только после изучения проекта. Не спрашивай то, на что уже можно ответить из файлов репозитория.
8. После вопросов остановись и дождись явного разрешения на реализацию.
9. Не выполняй `git push`, merge, удаление веток, массовое перемещение файлов или другие необратимые действия без отдельного разрешения.

## Главная цель продукта

VivAtlas должен отвечать на четыре вопроса:

1. Что есть в наших репозиториях?
2. Что делает каждый инструмент?
3. Какой инструмент или цепочка инструментов лучше подходят для конкретной задачи?
4. Что появилось, изменилось, устарело, конфликтует или требует внимания?

Программа должна работать как единый персональный диспетчер программных возможностей для **ChatGPT и Claude Code**.

# 1. Неподвижные продуктовые принципы

## 1.1. Структура Git — источник истины

В наших GitHub- и Gitea-репозиториях уже существует собственная поддерживаемая структура разделов и директорий.

Обязательные правила:

- не придумывать собственную обязательную иерархию поверх Git;
- не переносить файлы и папки автоматически;
- не переименовывать существующие разделы;
- не раскладывать инструменты по автоматически созданным директориям;
- показывать физическую структуру репозитория такой, какая она есть;
- использовать путь как источник метаданных и тегов, но не как повод менять структуру;
- любые предложения о реорганизации показывать только как рекомендацию;
- любое физическое изменение Git выполнять только после явного подтверждения пользователя.

Формула продукта:

```text
Физическая структура Git = источник истины
Пользовательские разделы = дополнительный виртуальный слой
Теги = дополнительный поисковый слой
AI = помощник, а не распорядитель
```

## 1.2. Три независимых уровня организации

### Физическая структура

Настоящие директории GitHub, Gitea и локальных репозиториев. Они отображаются без преобразований.

### Пользовательские разделы

Виртуальные папки внутри VivAtlas. Пользователь сам:

- создаёт разделы;
- создаёт вложенные разделы;
- переименовывает их;
- меняет порядок;
- назначает значки;
- добавляет один инструмент в несколько разделов;
- архивирует или скрывает разделы.

Пользовательские разделы не меняют Git.

### Умные разделы

Сохранённые фильтры, например:

- новые за 30 дней;
- плохо документированные;
- поддерживают Claude Code;
- работают с PDF;
- требуют внешнего API;
- не обновлялись больше года;
- имеют повышенный риск.

Пользователь сам определяет правила умных разделов.

## 1.3. Мобильная версия обязательна

Мобильная версия не является будущим дополнением. Она входит в основной продукт.

Первая версия должна быть полноценным адаптивным **PWA**:

- Android;
- iPhone и iPad;
- установка на главный экран;
- самостоятельный запуск;
- адаптивный интерфейс;
- базовый offline-кэш каталога;
- синхронизация изменений после восстановления сети;
- push-уведомления или подготовленная архитектура уведомлений;
- единый backend и единая база с настольной версией.

Не проектируй desktop-only интерфейс с последующим механическим уменьшением. Сразу продумай отдельные мобильные сценарии.

## 1.4. Автотеги обязательны

Автотеги должны быть одной из центральных возможностей MVP.

Источники тегов:

- имя инструмента;
- путь в репозитории;
- `SKILL.md` и `skill.md`;
- `README.md`;
- `CLAUDE.md`, `AGENTS.md`, `GEMINI.md`;
- YAML frontmatter;
- `package.json`, `pyproject.toml`, `requirements.txt`;
- зависимости;
- названия функций, команд и entrypoints;
- примеры использования;
- безопасный AI-анализ содержимого.

Разделяй теги на:

- системные;
- определённые из пути и файлов;
- AI-теги;
- пользовательские.

Для каждого тега сохраняй:

- источник;
- confidence;
- дату назначения;
- модель или правило, которое его создало;
- признак ручного подтверждения;
- признак ручного запрета.

Ручные действия имеют абсолютный приоритет:

- удалённый пользователем автотег не должен возвращаться после следующего сканирования;
- ручной тег нельзя перезаписывать;
- низкоуверенные теги должны попадать в предложения, а не назначаться автоматически;
- пороги автоприменения должны настраиваться.

# 5. Зеркала GitHub и Gitea

Одинаковый инструмент в GitHub и Gitea не должен отображаться двумя карточками.

Нужна каноническая запись с несколькими источниками.

Логика сопоставления должна учитывать:

- явно заданный `collection` или mirror-group;
- repository identity;
- относительный путь;
- manifest metadata;
- remote URLs;
- commit ancestry;
- content hash.

При расхождении версий показывать конфликт:

```text
Gitea опережает GitHub на 3 коммита.
Последний общий commit: <sha>
```

Никогда не синхронизировать зеркала автоматически без отдельной настройки и подтверждения.

# 6. Поиск и рекомендации

Поддержать одновременно:

- полнотекстовый поиск;
- фильтры;
- поиск по пути;
- поиск по тегам;
- смысловой multilingual-поиск на русском и английском;
- поиск по естественному описанию задачи;
- сравнение похожих инструментов;
- построение цепочки инструментов.

Пример запроса:

> Мне нужно взять большой PDF, извлечь таблицы и сделать аккуратный Excel.

Ответ должен содержать:

1. Лучший вариант.
2. До двух альтернатив.
3. Причины выбора.
4. Ограничения каждого варианта.
5. При необходимости — последовательность из нескольких инструментов.
6. Почему очевидные, но менее подходящие инструменты не выбраны.
7. Confidence рекомендации.
8. Признак, основана ли рекомендация на документации, тегах, истории использования или AI-инференсе.

Если подходящего инструмента нет, система обязана сказать об этом прямо и предложить:

- комбинацию существующих инструментов;
- расширение ближайшего инструмента;
- создание нового Skill;
- поиск внешнего решения.

Не придумывать отсутствующие возможности.

# 7. Создание разделов и директорий

В интерфейсе пользователь должен явно выбирать один из трёх вариантов:

```text
1. Пользовательский раздел
   Не изменяет Git.

2. Умный раздел
   Автоматически заполняется по фильтрам.

3. Директория в репозитории
   Создаёт физическую папку через Git-операцию.
```

Для физической директории поддержать:

- выбор провайдера;
- выбор репозитория;
- выбор ветки;
- создание вложенного пути;
- добавление `.gitkeep`, README или шаблона инструмента;
- предварительный diff;
- новую ветку;
- commit;
- pull request;
- прямой commit только при явном разрешении.

Перед записью показывать понятный preview:

```text
Будет создано:
skills/pdf/new-tool/
skills/pdf/new-tool/SKILL.md
skills/pdf/new-tool/agents/openai.yaml

Репозиторий: Gitea / tools
Ветка: feature/add-new-pdf-tool
Способ: Pull Request
```

AI и автотеги никогда не должны сами перемещать или удалять файлы.

# 8. Разрешения и безопасность

Для каждого источника должна быть отдельная матрица разрешений:

```yaml
permissions:
  read: true
  create_directories: true
  create_files: true
  move_files: false
  delete_files: false
  direct_commit: false
  create_pull_request: true
```

Обязательные меры:

- read-only по умолчанию;
- write-токен подключается отдельно;
- explicit confirmation для записи;
- дополнительное подтверждение для destructive operations;
- исключение `.env`, private keys, certificates, credentials и secret-файлов;
- secret scanning до передачи фрагментов внешнему AI;
- repository content считать недоверенным;
- инструкции внутри индексируемых файлов не исполнять;
- не запускать shell-команды из найденных файлов;
- журналировать операции;
- не отправлять приватный код внешнему AI без явной настройки;
- поддержать режим без внешнего AI;
- подготовить поддержку локальных моделей, например Ollama.

Риск артефакта классифицировать минимум как:

- read-only;
- modifies-files;
- external-network;
- credentials-access;
- shell-execution;
- destructive.

# 10. API и MCP

Нужны единый REST API и один MCP-сервер для ChatGPT и Claude Code.

Минимальные MCP tools:

```text
search_artifacts
recommend_artifacts
get_artifact
compare_artifacts
build_workflow
list_repository_tree
list_recent_changes
find_duplicates
find_stale_artifacts
find_poorly_documented
list_tags
list_sections
record_usage

```

Write-tools держать отдельно и отключёнными по умолчанию:

```text

prepare_move
```

Каждый write-tool должен возвращать preview и требовать отдельного подтверждения перед фактическим выполнением.

MCP-ответы должны быть компактными, структурированными и пригодными для дальнейшего reasoning ChatGPT/Claude.

# 11. Предпочтительная техническая архитектура

Сначала проверь существующий стек проекта. Не создавай второй backend или frontend, если уже есть пригодная архитектура.

Если проект не задаёт иной зрелый стек, предпочитай:

## Backend

- Python 3.12+;
- FastAPI;
- Pydantic v2;
- SQLAlchemy 2;
- Alembic;
- отдельный scanner/worker;
- официальный или поддерживаемый MCP SDK;
- GitHub API и Gitea API с абстракцией provider adapter.

## Frontend

- React;
- TypeScript;
- Vite;
- PWA/Service Worker;
- responsive mobile-first layout;
- доступность клавиатурой и screen reader;
- без жёсткой привязки UI к конкретному Git-провайдеру.

## Database

- PostgreSQL;
- pgvector для semantic search;
- SQLite допустим только для локального development/test режима;
- полнотекстовый поиск PostgreSQL или отдельный поисковый адаптер.

## Deployment

- Docker Compose;
- отдельные сервисы API, web, worker и database;
- health checks;
- migrations;
- `.env.example` без секретов;
- production и development profiles.

Архитектура должна позволять заменить:

- embedding provider;
- LLM provider;
- Git provider;
- search backend;
- background job runner.

Не делать fork Karakeep и не копировать его код. Можно использовать только продуктовые идеи: карточки, теги, поиск, коллекции, правила и массовые действия.

# 12. Инкрементальное сканирование

Не пересканировать весь репозиторий без необходимости.

Продумать:

- last scanned commit;
- commit diff;
- content hash;
- renamed/moved files;
- deleted files;
- parser version;
- AI-analysis version;
- tag-rule version;
- reindex queue;
- retry и error states;
- rate limits GitHub/Gitea;
- webhook verification.

После сканирования формировать changelog:

```text
+ новые
~ обновлённые
- удалённые
↪ перемещённые
! конфликты зеркал
⚠ проблемы документации
```

# 13. Оценка качества

Оценка должна строиться из объяснимых компонентов, а не из магического одного числа.

Минимальные показатели:

- documentation completeness;
- description clarity;
- freshness;
- installability;
- test evidence;
- dependency health;
- platform compatibility;
- security risk;
- recommendation confidence.

Показывай причины оценки.

Не делай вывод «работает», если инструмент не был реально проверен. Используй статусы:

- discovered;
- documented;
- statically-validated;
- tests-present;
- tests-passed;
- user-verified;
- deprecated;
- broken;
- unknown.

# 16. Правила реализации после подтверждения

Когда пользователь одобрит план:

1. Работай по согласованным этапам.
2. Перед каждым крупным этапом кратко указывай, что меняешь.
3. Не меняй несвязанные файлы.
4. Делай минимальные, осмысленные изменения.
5. Сохраняй backward compatibility, где это разумно.
6. После каждого этапа запускай относящиеся к нему тесты, lint и type-check.
7. Исправляй причины ошибок, а не скрывай их отключением проверок.
8. Не добавляй mock-реализации в production path без явной маркировки.
9. Не выдавай статический keyword search за semantic search.
10. Не выдавай наличие тестов за успешное прохождение тестов.
11. Не утверждай, что мобильная версия готова, пока не проверены реальные mobile breakpoints, installability PWA и offline state.
12. Обновляй документацию вместе с кодом.
13. Веди changelog выполненных этапов.
14. В конце каждого этапа сообщай:
    - что готово;
    - что проверено;
    - какие команды тестов запускались;
    - что осталось;
    - какие ограничения обнаружены.

# 17. Критерий конечного успеха

Сценарий должен работать от начала до конца:

1. Пользователь подключает приватные Gitea и GitHub.
2. VivAtlas сканирует существующую структуру, не изменяя её.
3. Программа находит Skills, Claude-команды, агентов, MCP, плагины и скрипты.
4. Создаёт понятные карточки и автотеги.
5. Показывает реальное дерево репозитория.
6. Пользователь создаёт собственные виртуальные разделы.
7. Пользователь с телефона пишет:

   > Мне нужно проанализировать PDF, извлечь таблицы и сделать Excel.

8. Система предлагает подходящий инструмент или цепочку, объясняет выбор и ограничения.
9. Те же данные доступны ChatGPT и Claude Code через MCP.
10. После нового push обновляются только изменившиеся элементы.
11. Пользователь может создать новую физическую директорию через preview, ветку и pull request.
12. Ни AI, ни автотеги не меняют структуру Git самостоятельно.

Начни с аудита текущего репозитория и подготовь материалы из раздела **«Что требуется от тебя до начала реализации»**. Реализацию пока не начинай.
